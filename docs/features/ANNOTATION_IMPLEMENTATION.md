# Text Annotation Implementation Summary

This document provides an overview of the text annotation feature implementation for YSimulator.

## Overview

Every post and comment generated by LLM or rule-based agents is now automatically annotated with:
- **Hashtags**: Extracted and stored in `hashtags` and `post_hashtags` tables
- **User Mentions**: Validated against existing users and stored in `mentions` table
- **Sentiment**: Computed using VADER and stored in `post_sentiment` table (per topic)
- **Toxicity**: Computed using the Perspective API (when an API key is supplied) or the
  **detoxify** library locally (when no API key is supplied), and stored in the
  `post_toxicity` table

## Configuration

Add these flags to your `simulation_config.json` under the `"simulation"` section:

```json
{
  "simulation": {
    ...
    "enable_sentiment": true,
    "enable_toxicity": false,
    "perspective_api_key": null
  }
}
```

- `enable_sentiment`: Set to `true` to enable sentiment analysis (default: false)
- `enable_toxicity`: Set to `true` to enable toxicity analysis (default: false)
- `perspective_api_key`: Your Perspective API key. When omitted or set to `null`, toxicity
  scoring falls back to the local **detoxify** model automatically — no API key is required.

## Architecture

### Client-Side (YClient)

**Text Annotation Module** (`YSimulator/YClient/text_support/text_annotator.py`)
- `annotate_text()`: Main function that coordinates all annotations
- Uses existing functions from `cleaning.py` and `annotations.py`
- Returns dict with hashtags, mentions, sentiment, and toxicity data

**Toxicity Back-ends** (`YSimulator/YClient/text_support/annotations.py`)

| Condition | Back-end used |
|-----------|--------------|
| `perspective_api_key` is set | Perspective API (remote, requires internet) |
| `perspective_api_key` is `None` | detoxify `"original"` model (local, no API key needed) |

The detoxify model is loaded lazily on first use and cached for the lifetime of the process.
A `logging.getLogger(__name__)` logger records:
- `INFO` when the detoxify model is first initialised
- `DEBUG` when the detoxify fallback path is taken
- `WARNING` on any scoring failure (either back-end), with the exception message

**Action Processing**
- All posts/comments are annotated before being sent to the server
- Annotations are attached to `ActionDTO` objects via the `annotations` field
- Both LLM-generated and rule-based content are annotated

### Server-Side (YServer)

**Database Methods** (`YSimulator/YServer/classes/service layer.py`)
- `add_or_get_hashtag(hashtag_text)`: Gets or creates hashtag entries
- `add_post_hashtag(post_id, hashtag_id)`: Links hashtags to posts
- `get_user_by_username(username)`: Validates mentioned users
- `add_mention(mention_data)`: Creates mention entries
- `add_post_sentiment(sentiment_data)`: Stores sentiment per topic
- `add_post_toxicity(toxicity_data)`: Stores toxicity scores

**Annotation Processing** (`YSimulator/YServer/server.py`)
- `_process_annotations()`: Main method that processes all annotation types
- Called when POST or COMMENT actions are received in `submit_actions()`
- Handles hashtag insertion, mention validation, sentiment storage, and toxicity storage

## Database Schema

### Hashtags
```
hashtags (id, hashtag)
post_hashtags (id, post_id, hashtag_id)
```

### Mentions
```
mentions (id, user_id, post_id, round, answered)
```
- `user_id`: The user being mentioned
- `post_id`: The post/comment containing the mention
- `answered`: Flag for tracking if the mention was responded to (default: 0)

### Sentiment
```
post_sentiment (
  id, post_id, user_id, topic_id, round,
  neg, pos, neu, compound,
  sentiment_parent,
  is_post, is_comment, is_reaction
)
```
- One entry per topic associated with the post/comment
- `sentiment_parent`: Compound sentiment of parent post (for comments only)
- Flags indicate whether it's a post, comment, or reaction

### Toxicity
```
post_toxicity (
  id, post_id,
  toxicity, severe_toxicity, identity_attack,
  insult, profanity, threat,
  sexually_explicit, flirtation
)
```
- All scores are stored regardless of which back-end produced them
- Only one entry per post/comment
- Note: detoxify does not model `flirtation`; that field is stored as `0.0`

```json
{
  "simulation": {
    "enable_sentiment": true,
    "enable_toxicity": false,

  }
```

## Code Structure

### Key Files Modified

1. **YSimulator/YClient/text_support/annotations.py**
   - `detoxify_toxicity()`: Local toxicity scoring via detoxify
   - `_get_detoxify_model()`: Lazy-initialised, cached Detoxify model
   - `toxicity()`: Delegates to Perspective API or detoxify depending on `api_key`
   - Module-level logger (`logging.getLogger(__name__)`)

2. **YSimulator/YClient/text_support/text_annotator.py** (NEW)
   - Main annotation orchestration
   - Calls `toxicity()` whenever `enable_toxicity=True`, regardless of whether an
     API key is present

3. **YSimulator/YClient/client.py**
   - Load annotation config in `__init__`
   - `_annotate_action_content()`: Helper method for rule-based actions
   - `_gather_pending_llm_posts()`: Annotate LLM posts
   - `_gather_pending_llm_reactions()`: Annotate LLM comments
   - All rule-based action handlers now annotate content

4. **YSimulator/YClient/classes/ray_models.py**
   - Added `annotations` field to `ActionDTO`

5. **YSimulator/YServer/classes/service layer.py**
   - Added 6 new database methods for annotation storage

6. **YSimulator/YServer/server.py**
   - `_process_annotations()`: Process all annotation types
   - Modified `submit_actions()` to call annotation processing

7. **example/simulation_config.json**
   - Added annotation configuration flags

## Dependencies

The annotation feature requires:
- `nltk>=3.8.0` (for VADER sentiment analysis)
- `detoxify>=0.5.2` (for local toxicity scoring without an API key)
- `perspective` (optional — for Perspective API toxicity analysis)

Install via:
```bash
pip install nltk detoxify perspective
python -c "import nltk; nltk.download('vader_lexicon')"
```

## Performance Considerations

- **Sentiment Analysis**: Fast, runs locally with VADER
- **Toxicity (detoxify)**: Runs locally; first call incurs a one-time model-load cost
  (~1–2 s on CPU). Subsequent calls reuse the cached model and are faster.
- **Toxicity (Perspective API)**: Requires API calls to Perspective API (slower, has
  rate limits and requires internet access)
- **Database Operations**: Each annotation type requires separate database inserts
- For high-volume simulations, consider disabling toxicity or batching database operations

## Troubleshooting

**Issue**: `ModuleNotFoundError: No module named 'nltk'`
**Solution**: Install NLTK and download VADER lexicon
```bash
pip install nltk
python -c "import nltk; nltk.download('vader_lexicon')"
```

**Issue**: `ModuleNotFoundError: No module named 'detoxify'`
**Solution**: Install detoxify
```bash
pip install detoxify>=0.5.2
```

**Issue**: Toxicity scores are all empty (`{}`)
**Solution**: Check that:
1. `enable_toxicity` is set to `true`
2. If using the Perspective API, `perspective_api_key` contains a valid API key and the
   API is accessible from your network
3. Check application logs for `WARNING` messages from
   `YSimulator.YClient.text_support.annotations` which report the exact failure reason

**Issue**: First toxicity call is slow when no API key is set
**Solution**: This is expected — the detoxify model is loaded from disk on the first call
and cached for all subsequent calls. No action required.

**Issue**: Mentions not being stored
**Solution**: Ensure the mentioned users exist in the `user_mgmt` table with matching
usernames
